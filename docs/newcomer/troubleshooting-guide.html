Build and Compilation Issues

Issue: Maven Build Fails with Dependency Errors

Symptoms: 
[ERROR] Failed to execute goal on project service-billing: 
Could not resolve dependencies for project com.embrix.core:service-billing:jar:3.1.9-SNAPSHOT
Causes:
  • Modules built in wrong order
  • Missing artifacts in local Maven repository
  • Version mismatch between modules
Solutions:
  1. Build in correct order:
# Foundation first
cd common && mvn clean install
cd ../engine && mvn clean install
cd ../gateway-common && mvn clean install
cd ../oms-component && mvn clean install
cd ../jobs-common && mvn clean install

# Then services
cd ../core/service-billing && mvn clean install
  1. Clean Maven cache:
mvn dependency:purge-local-repository
mvn clean install
  1. Check version consistency:
# Verify all modules use same version
grep -r "3.1.9-SNAPSHOT" pom.xml

Issue: Groovy Compilation Errors

Symptoms: 
[ERROR] Failed to execute goal org.codehaus.gmavenplus:gmavenplus-plugin:1.6:compile
Causes:
  • Groovy version mismatch
  • Mixed Java/Groovy source issues
  • IDE and Maven using different Groovy versions
Solutions:
  1. Verify Groovy version:
<groovy.version>2.4.15</groovy.version>
  1. Clean and rebuild:
mvn clean compile -U
  1. Check IDE Groovy plugin:
  • IntelliJ: Settings ‚Üí Plugins ‚Üí Groovy (ensure version 2.4.15)
  • Eclipse: Install Groovy Eclipse plugin

Issue: JOOQ Code Generation Fails

Symptoms: 
[ERROR] Error while generating jOOQ code: 
org.jooq.exception.DataAccessException: SQL [SELECT ...]; ERROR: relation "account" does not exist
Causes:
  • Database not initialized
  • Flyway migrations not run
  • Wrong database connection
Solutions:
  1. Run Flyway migrations first:
mvn flyway:migrate -Dspring.profiles.active=dev
  1. Verify database connection:
psql -U omsadmin -d omsdevdb -c "\dt"
  1. Generate JOOQ code:
cd engine
mvn clean compile -Pjooq-codegen

Database Connection Problems

Issue: Cannot Connect to PostgreSQL

Symptoms: 
org.postgresql.util.PSQLException: Connection to localhost:5432 refused
Solutions:
  1. Verify PostgreSQL is running:
docker ps | grep postgres
  1. Start PostgreSQL if not running:
docker run -d --name postgres \
  -p 5432:5432 \
  -e POSTGRES_USER=omsadmin \
  -e POSTGRES_PASSWORD=omsadmin \
  -e POSTGRES_DB=omsdevdb \
  postgres:10.5
  1. Check port availability:
netstat -an | grep 5432
  1. Test connection:
psql -h localhost -U omsadmin -d omsdevdb

Issue: Flyway Migration Failures

Symptoms: 
FlywayException: Validate failed: 
Migration checksum mismatch for migration version 1.0
Causes:
  • Migration files modified after execution
  • Database schema manually changed
  • Version conflicts
Solutions:
  1. Check migration history:
mvn flyway:info -Dspring.profiles.active=dev
  1. Repair flyway_schema_history:
mvn flyway:repair -Dspring.profiles.active=dev
  1. Baseline and re-migrate (WARNING: only for dev):
mvn flyway:clean -Dspring.profiles.active=dev
mvn flyway:migrate -Dspring.profiles.active=dev
  1. For production, create repair migration:
-- V100__repair_schema.sql
-- Add specific fixes here

Issue: Database Connection Pool Exhausted

Symptoms: 
HikariPool-1 - Connection is not available, request timed out after 30000ms
Causes:
  • Too many open connections
  • Long-running queries
  • Connection leaks
Solutions:
  1. Check active connections:
SELECT 
    count(*),
    state,
    application_name
FROM pg_stat_activity
GROUP BY state, application_name;
  1. Increase pool size:
spring:
  datasource:
    hikari:
      maximum-pool-size: 20
      minimum-idle: 5
  1. Find long-running queries:
SELECT 
    pid,
    now() - pg_stat_activity.query_start AS duration,
    query
FROM pg_stat_activity
WHERE state = 'active'
ORDER BY duration DESC;
  1. Kill problematic connection:
SELECT pg_terminate_backend(pid);

Service Startup Failures

Issue: Spring Boot Application Won't Start

Symptoms: 
APPLICATION FAILED TO START
Description: Failed to configure a DataSource: 'url' attribute is not specified
Causes:
  • Missing environment variables
  • Vault connection failure
  • Application.yml misconfiguration
Solutions:
  1. Check required environment variables:
export DB_URL=jdbc:postgresql://localhost:5432/omsdevdb
export DB_USER=omsadmin
export DB_PASSWORD=omsadmin
export VAULT_URI=http://localhost:8200
export VAULT_TOKEN=myroot
export AMQ_BROKER_URL=tcp://localhost:61616
  1. Verify Vault connection:
curl -H "X-Vault-Token: myroot" http://localhost:8200/v1/sys/health
  1. Use application-dev.yml for local development:
spring:
  datasource:
    url: jdbc:postgresql://localhost:5432/omsdevdb
    username: omsadmin
    password: omsadmin
  1. Start with specific profile:
mvn spring-boot:run -Dspring.profiles.active=dev

Issue: Port Already in Use

Symptoms: 
Web server failed to start. Port 8080 was already in use.
Solutions:
  1. Find process using port:
# Windows
netstat -ano | findstr :8080
taskkill /PID <PID> /F

# Linux/Mac
lsof -ti:8080
kill -9 <PID>
  1. Change service port:
server:
  port: 8081
  1. Use random port for tests:
server:
  port: 0  # Random available port

Issue: Bean Creation Failures

Symptoms: 
Error creating bean with name 'customerHub': Unsatisfied dependency expressed through field 'accountService'
Causes:
  • Circular dependencies
  • Missing @Component annotations
  • Configuration class not scanned
Solutions:
  1. Check component scanning:
@SpringBootApplication
@ComponentScan(basePackages = {
    "com.embrix.core.engine",
    "com.embrix.core.common"
})
public class Application {
    // ...
}
  1. Break circular dependency:
// Use @Lazy injection
@Autowired
@Lazy
private AccountService accountService;
  1. Verify @Component/@Service annotations:
@Service
class CustomerHub {
    // ...
}

ActiveMQ and Messaging Issues

Issue: Cannot Connect to ActiveMQ

Symptoms: 
javax.jms.JMSException: Could not connect to broker URL: tcp://localhost:61616
Solutions:
  1. Verify ActiveMQ is running:
docker ps | grep activemq
  1. Start ActiveMQ:
docker run -d --name activemq \
  -p 61616:61616 \
  -p 8161:8161 \
  webcenter/activemq:5.14.3
  1. Check ActiveMQ console:
http://localhost:8161/admin
Username: admin
Password: admin
  1. Verify broker URL:
spring:
  activemq:
    broker-url: tcp://localhost:61616
    user: admin
    password: admin

Issue: Messages Not Being Consumed

Symptoms:
  • Messages accumulate in queue
  • No consumer activity
  • Orders stuck in CREATED status
Causes:
  • Consumer not running
  • Message selector mismatch
  • Consumer exceptions
Solutions:
  1. Check queue depth in ActiveMQ console:
http://localhost:8161/admin/queues.jsp
  1. Verify consumer is registered:
@JmsListener(destination = "OMS")
public void receiveMessage(String message) {
    log.info("Received: {}", message);
}
  1. Check consumer logs for exceptions:
tail -f logs/application.log | grep "JmsListener"
  1. Manually browse queue:
# Use JMX or ActiveMQ browse feature
curl -u admin:admin http://localhost:8161/api/message/OMS?type=queue

Issue: Message Redelivery Loop

Symptoms: 
Message redelivered 6 times, moving to DLQ
Causes:
  • Consumer throwing exceptions
  • Transaction rollback
  • Invalid message format
Solutions:
  1. Check Dead Letter Queue:
Queue: ActiveMQ.DLQ
  1. Review error logs:
grep "MessageListenerException" logs/application.log
  1. Fix consumer logic:
@JmsListener(destination = "OMS")
void processOrder(String message) {
    try {
        // Process message
    } catch (Exception e) {
        log.error("Failed to process: {}", message, e)
        // Don't rethrow - acknowledge message
    }
}
  1. Configure redelivery policy:
application:
  queues:
    redeliveryPolicy:
      maxRetries: 3
      delay: 1000
      multiplier: 2

GraphQL API Errors

Issue: GraphQL 404 Not Found

Symptoms: 
404 Not Found: GET /graphql
Causes:
  • Service not started
  • Wrong endpoint URL
  • GraphQL not enabled
Solutions:
  1. Verify service is running:
curl http://localhost:8080/actuator/health
  1. Check GraphQL endpoint:
curl -X POST http://localhost:8080/graphql \
  -H "Content-Type: application/json" \
  -d '{"query": "{ __typename }"}'
  1. Access GraphiQL playground:
http://localhost:8080/graphiql
  1. Verify GraphQL dependency:
<dependency>
    <groupId>com.graphql-java</groupId>
    <artifactId>graphql-spring-boot-starter</artifactId>
</dependency>

Issue: GraphQL Validation Errors

Symptoms: 
{
  "errors": [
    {
      "message": "Validation error of type FieldUndefined: Field 'accountId' is undefined"
    }
  ]
}
Causes:
  • Field doesn't exist in schema
  • Typo in field name
  • Schema not reloaded
Solutions:
  1. Check GraphQL schema:
# schema.graphqls
type Account {
    accountId: String!
    accountName: String
}
  1. Verify field resolver:
class AccountResolver {
    String accountId(Account account) {
        return account.accountId
    }
}
  1. Reload schema (dev mode):
# Restart service
mvn spring-boot:run

Issue: GraphQL Performance Issues

Symptoms:
  • Slow query execution
  • Timeout errors
  • N+1 query problem
Solutions:
  1. Use DataLoader for batch loading:
@Bean
public DataLoader<String, Account> accountDataLoader() {
    return DataLoader.newDataLoader(accountIds -> 
        accountService.findByIds(accountIds)
    );
}
  1. Limit query depth:
@Bean
public GraphQLFieldDefinition rootQuery() {
    return GraphQLFieldDefinition.newFieldDefinition()
        .type(queryType)
        .dataFetcher(env -> {
            int depth = env.getField().getSelectionSet().getFields().size();
            if (depth > 5) {
                throw new RuntimeException("Query too complex");
            }
            // ...
        });
}
  1. Add pagination:
query {
  searchAccounts(pagination: { page: 0, size: 20 }) {
    content { ... }
  }
}

Vault and Secrets Management

Issue: Vault Connection Refused

Symptoms: 
VaultException: Status 500, Connection refused: localhost/127.0.0.1:8200
Solutions:
  1. Start Vault in dev mode:
docker run -d --name vault \
  -p 8200:8200 \
  -e VAULT_DEV_ROOT_TOKEN_ID=myroot \
  vault:latest
  1. Verify Vault is running:
curl http://localhost:8200/v1/sys/health
  1. Set environment variables:
export VAULT_URI=http://localhost:8200
export VAULT_TOKEN=myroot

Issue: Vault Secrets Not Found

Symptoms: 
VaultException: Status 404, Path 'secret/ctg-oms/dev' not found
Solutions:
  1. Write secrets to Vault:
# Using Vault CLI
vault kv put secret/ctg-oms/dev \
  db.url=jdbc:postgresql://localhost:5432/omsdevdb \
  db.username=omsadmin \
  db.password=omsadmin

# Using curl
curl -H "X-Vault-Token: myroot" \
  -X POST \
  -d '{"db.url":"jdbc:postgresql://localhost:5432/omsdevdb"}' \
  http://localhost:8200/v1/secret/ctg-oms/dev
  1. Verify secret path:
vault kv list secret/
vault kv get secret/ctg-oms/dev
  1. Check application configuration:
spring:
  cloud:
    vault:
      uri: ${VAULT_URI:http://localhost:8200}
      token: ${VAULT_TOKEN}
      application-name: ctg-oms

Performance Issues

Issue: Slow Database Queries

Symptoms:
  • Long response times
  • High CPU usage
  • Database connection timeouts
Solutions:
  1. Identify slow queries:
SELECT 
    query,
    mean_exec_time,
    calls
FROM pg_stat_statements
ORDER BY mean_exec_time DESC
LIMIT 10;
  1. Analyze query plan:
EXPLAIN ANALYZE
SELECT * FROM account WHERE account_name = 'TechCorp';
  1. Add missing indexes:
CREATE INDEX idx_account_name ON account(account_name);
CREATE INDEX idx_order_account_id ON order(account_id);
CREATE INDEX idx_invoice_account_id ON invoice(account_id);
  1. Optimize JOOQ queries:
// Bad: Fetches all fields
def accounts = dsl.selectFrom(ACCOUNT).fetch()

// Good: Select specific fields
def accounts = dsl.select(
    ACCOUNT.ACCOUNT_ID,
    ACCOUNT.ACCOUNT_NAME
).from(ACCOUNT).fetch()

Issue: High Memory Usage

Symptoms: 
java.lang.OutOfMemoryError: Java heap space
Solutions:
  1. Increase heap size:
export JAVA_OPTS="-Xms512m -Xmx2048m"
mvn spring-boot:run
  1. Profile memory usage:
jmap -heap <PID>
jstat -gc <PID> 1000
  1. Check for memory leaks:
# Generate heap dump
jmap -dump:live,format=b,file=heap.bin <PID>

# Analyze with Eclipse MAT or VisualVM
  1. Optimize list operations:
// Bad: Loads everything into memory
def allAccounts = accountService.findAll()

// Good: Use pagination
def page = accountService.findAll(PageRequest.of(0, 100))

Multi-Tenant Configuration Problems

Issue: Wrong Tenant Data Returned

Symptoms:
  • User sees data from different tenant
  • Orders appear in wrong tenant
Causes:
  • Tenant context not set
  • Missing tenant filter in queries
  • Cache key doesn't include tenant
Solutions:
  1. Verify tenant context:
def tenantId = TenantContext.getCurrentTenant()
log.info("Current tenant: {}", tenantId)
  1. Add tenant filter to queries:
def accounts = dsl.selectFrom(ACCOUNT)
    .where(ACCOUNT.TENANT_ID.eq(tenantId))
    .fetch()
  1. Include tenant in cache keys:
@Cacheable(value = "accounts", key = "#tenantId + ':' + #accountId")
def findAccount(String tenantId, String accountId) {
    // ...
}

Issue: Tenant-Specific Configuration Not Loading

Symptoms: 
Using default configuration instead of tenant-specific
Solutions:
  1. Verify Helm values exist:
ls core/service-billing/helm_values/urbanos/
  1. Check tenant configuration in database:
SELECT * FROM core_config.tenant WHERE tenant_id = 'TIDLT-100005';
  1. Verify environment variable:
echo $TENANT_ID
  1. Load tenant config explicitly:
@Value("\${tenant.id}")
String tenantId

@PostConstruct
void loadTenantConfig() {
    def config = tenantConfigService.load(tenantId)
    log.info("Loaded config for tenant: {}", tenantId)
}

Frontend Issues

Issue: React App Won't Start

Symptoms: 
Error: ENOSPC: System limit for number of file watchers reached
Solutions:
  1. Increase file watcher limit (Linux):
echo fs.inotify.max_user_watches=524288 | sudo tee -a /etc/sysctl.conf
sudo sysctl -p
  1. Clear node_modules and reinstall:
rm -rf node_modules package-lock.json
npm install
  1. Check Node.js version:
node --version  # Should be 12.x or higher
npm --version

Issue: Apollo Client Network Errors

Symptoms: 
Error: Network error: Failed to fetch
Solutions:
  1. Check backend is running:
curl http://localhost:8080/graphql
  1. Verify Apollo configuration:
const client = new ApolloClient({
  uri: process.env.REACT_APP_APOLLO_API_URL || 'http://localhost:8080/graphql',
  cache: new InMemoryCache()
});
  1. Check CORS configuration:
@Configuration
public class CorsConfig {
    @Bean
    public WebMvcConfigurer corsConfigurer() {
        return new WebMvcConfigurer() {
            @Override
            public void addCorsMappings(CorsRegistry registry) {
                registry.addMapping("/graphql")
                    .allowedOrigins("http://localhost:3000")
                    .allowedMethods("GET", "POST");
            }
        };
    }
}

Deployment and Infrastructure

Issue: Docker Container Won't Start

Symptoms: 
Error: Container exits immediately
Solutions:
  1. Check container logs:
docker logs <container-id>
docker logs --tail 100 postgres
  1. Verify port mapping:
docker ps
netstat -an | grep 5432
  1. Check resource limits:
docker stats
  1. Inspect container:
docker inspect <container-id>

Issue: Kubernetes Pod CrashLoopBackOff

Symptoms: 
NAME                     READY   STATUS             RESTARTS   AGE
service-billing-xxx      0/1     CrashLoopBackOff   5          5m
Solutions:
  1. Check pod logs:
kubectl logs service-billing-xxx
kubectl logs service-billing-xxx --previous
  1. Describe pod:
kubectl describe pod service-billing-xxx
  1. Check events:
kubectl get events --sort-by='.lastTimestamp'
  1. Verify ConfigMap and Secrets:
kubectl get configmap
kubectl get secret
kubectl describe configmap service-billing-config

Testing Failures

Issue: Integration Tests Fail

Symptoms: 
Tests run: 50, Failures: 10, Errors: 5, Skipped: 0
Solutions:
  1. Check test database:
docker ps | grep postgres-test
  1. Use Testcontainers:
@Testcontainers
class IntegrationTest {
    @Container
    static PostgreSQLContainer postgres = new PostgreSQLContainer("postgres:10.5")
        .withDatabaseName("test")
        .withUsername("test")
        .withPassword("test")
}
  1. Clean test data between tests:
@BeforeEach
void setup() {
    dsl.deleteFrom(ACCOUNT).execute()
    dsl.deleteFrom(ORDER).execute()
}
