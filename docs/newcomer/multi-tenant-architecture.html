Multi-Tenant Architecture - Embrix O2X

Executive Summary

Architecture Pattern: Deployment-Level Multi-Tenancy (also known as "Instance-per-Tenant" or "Dedicated Deployment") Each tenant gets:
  • ‚úÖ Dedicated Kubernetes deployment
  • ‚úÖ Dedicated database (coredb-{tenant-name})
  • ‚úÖ Dedicated service instances (core services, gateways)
  • ‚úÖ Shared infrastructure (Kubernetes cluster, AWS resources)

4. Gateway Multi-Tenant Configuration

Merchant Configuration Lookup

Gateways retrieve tenant-specific merchant configurations from shared core_config: 
// From TenantRepositoryServiceImpl
Tenant findByTenantId(String id, MerchantType gateway_type) {
    jdbcTemplate.query(
        "SELECT * FROM ${TABLE_TENANT_MERCHANT_ACCOUNTS} " +
        "WHERE id='${id}' AND type='${gateway_type.name()}' AND status='ACTIVE'",
        new TenantMerchantsRowMapper()
    )
}

Example: QuickBooks OAuth2 (Finance Gateway)

  1. Finance gateway receives request with tenantId=TIDLT-100005
  2. Queries core_config.tenant_merchants for finance gateway config
  3. Retrieves OAuth2 attributes from core_config.oauth2_attributes
  4. Fetches credentials from Vault: http://urbanos-vault-interface/
  5. Performs QuickBooks API call with tenant-specific credentials

5. Frontend Layer

No Tenant Selection in UI

Frontend applications (ui-core, selfcare) do not:
  • ‚ùå Pass tenant ID in headers

Advantages of This Architecture

‚úÖ Strong Data Isolation

  • Complete database separation
  • No risk of cross-tenant data leaks
  • Simplified compliance/auditing

‚úÖ Performance Isolation

  • One tenant's load doesn't impact others
  • Dedicated resources per tenant
  • Independent scaling

‚úÖ Customization Flexibility

  • Can deploy different versions per tenant
  • Tenant-specific configuration without code changes
  • Easy A/B testing for specific tenants

‚úÖ Simplified Development

  • No tenant_id filtering in business logic
  • No complex query interceptors
  • Easier to test (no tenant context mocking)

‚úÖ Independent Operations

  • Deploy/upgrade tenants independently
  • Rollback single tenant without affecting others
  • Different maintenance windows per tenant

Challenges & Considerations

‚ö†Ô∏è Infrastructure Cost

  • Higher resource usage (one deployment per tenant)
  • More Kubernetes pods to manage
  • Database licensing considerations

‚ö†Ô∏è Operational Complexity

  • More deployments to monitor
  • More CI/CD pipelines to maintain
  • Helm chart management per tenant

‚ö†Ô∏è Onboarding Process

    2. Helm values configuration
    3. Kubernetes deployment
    4. Vault setup
    5. DNS/ingress configuration

‚ö†Ô∏è Shared Resources

  • Shared Kubernetes cluster
  • Shared Redis/ElastiCache
  • Shared ActiveMQ broker

Multi-Tenant Onboarding Flow

Adding a New Tenant

1. Database Setup 
-- On shared RDS instance
CREATE DATABASE "coredb-newtenant";

-- Run migrations
flyway migrate -schemas=public,core_config
2. Tenant Configuration 
-- Insert into core_config.tenant
INSERT INTO core_config.tenant (id, name, vaulturi, vaultpath, licensekey)
VALUES ('TIDLT-100008', 'newtenant', 'http://vault', 'tenant/newtenant', 'license-key');

-- Configure merchants
INSERT INTO core_config.tenant_merchants (id, tenant_id, name, type, status, authtype)
VALUES ('TIDLT-100008', 'TIDLT-100008', 'QUICKBOOKS', 'FINANCE_GATEWAY', 'ACTIVE', 'OAUTH2');
3. Helm Configuration 
# Create helm values files
mkdir -p core/helm_values/newtenant
mkdir -p finance-gateway/helm_values/newtenant
mkdir -p payment-gateway/helm_values/newtenant
mkdir -p tax-gateway/helm_values/newtenant
# ... etc for all services
4. Deploy Services 
helm install newtenant-service-transactional ./core/service-transactional \
  -f core/helm_values/newtenant/service-transactional.yaml

helm install newtenant-finance-gateway ./finance-gateway \
  -f finance-gateway/helm_values/newtenant/finance-gateway.yaml
# ... etc
5. Vault Setup 
# Store tenant-specific secrets
vault kv put secret/tenant/newtenant/quickbooks \
  clientId=xxx \
  clientSecret=xxx \
  accessToken=xxx \
  refreshToken=xxx

Recommendations

For Current Architecture

  1. Document Tenant Onboarding Process
    • Create automated scripts for tenant provisioning

    • Standardize Helm values templates

    • Automate database setup
  2. Monitor Resource Usage
    • Track per-tenant resource consumption

    • Consider dedicated RDS instances for high-volume tenants

    • Implement resource quotas per tenant namespace
  3. Version Management
    • Maintain tenant-to-version mapping

    • Document upgrade procedures per tenant

    • Implement blue-green deployment for critical tenants
  4. Backup & DR
    • Per-tenant backup schedules

    • Document restore procedures

    • Test disaster recovery for each tenant

If Scaling to Many Tenants

Consider hybrid approach:
  • Tier 1 (Enterprise): Dedicated deployment (current approach)
  • Tier 2 (SMB): Shared deployment with row-level tenant_id filtering
  • Tier 3 (Free/Trial): Highly shared, limited resources
This would require:
  • Adding tenant_id columns to business tables
  • Implementing row-level security (Hibernate filters, JPA interceptors)
  • Tenant context management (ThreadLocal, request scope)
  • Multi-tenancy middleware layer

Conclusion

The current architecture provides strong tenant isolation through deployment-level multi-tenancy, which is appropriate for:
  • ‚úÖ Small to medium number of tenants (< 50)
  • ‚úÖ Enterprise customers requiring dedicated resources
  • ‚úÖ Compliance requirements needing strict data separation
  • ‚úÖ Customers with varying SLA requirements
The trade-off is operational complexity and infrastructure cost, which scales linearly with tenant count.
