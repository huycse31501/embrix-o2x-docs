Embrix O2X Platform - Complete System Overview

Version: 3.1.9-SNAPSHOT Last Updated: February 2026 Purpose: Comprehensive bird's-eye view of the entire system architecture

Executive Summary

Embrix O2X (Order-to-X) is an enterprise-grade BSS/OSS platform for telecommunications and service providers. It handles the complete order-to-cash lifecycle: from order entry through provisioning, usage tracking, billing, invoicing, payment processing, and revenue recognition. Platform Scale:
  • 23 Services: 9 core services + 6 gateways + 5 foundation libraries + 3 additional
  • 50+ Database Tables: Organized across 8 PostgreSQL schemas
  • 11 Business Hubs: Domain-driven business logic organization
  • 7 Production Tenants: Currently serving real customers
  • Multi-tenant Architecture: Database-per-tenant with complete isolation

Security Architecture

Authentication

OAuth2 Flow:
  1. Client requests token from crm_gateway:8080/oauth/token
  2. Validates client credentials (from Vault)
  3. Issues JWT token with tenant and permissions
  4. Client includes token in Authorization header
Token Structure: 
{
  "sub": "client-id",
  "tenant_id": "TIDLT-100008",
  "scopes": ["all"],
  "permissions": ["CREATE_ORDER", "VIEW_INVOICE"],
  "exp": 1234567890
}

Authorization

Permission-Based:
  • Permissions checked via JWT claims
  • Method-level security: @PreAuthorize("hasPermission('CREATE_ORDER')")
  • Tenant context extracted from token

Secrets Management

HashiCorp Vault:
  • Secret paths: secret/{application}/{profile}
  • Stored secrets:
    • Database credentials

    • ActiveMQ credentials

    • OAuth client secrets

    • JWT signing keys

Key Design Patterns

Shared Engine Pattern

Principle: Centralize business logic Implementation: All services depend on engine library Benefit: Single source of truth, consistency

Event-Driven Architecture

Principle: Asynchronous communication Implementation: ActiveMQ message queues Benefit: Resilience, scalability, decoupling

Gateway Pattern

Principle: Isolate external dependencies Implementation: Gateway services normalize external APIs Benefit: Change external systems without breaking core logic

Hub-Based Organization

Principle: Domain-driven design Implementation: Engine organized into business hubs Benefit: Clear responsibility boundaries, maintainability

Multi-Tenant Deployment

Principle: Complete tenant isolation Implementation: Database-per-tenant, deployment-per-tenant Benefit: Data security, customization flexibility

TENANT_ID=TIDLT-100008
SPRING_PROFILES_ACTIVE=dev

# Database

DATABASE_USERNAME=omsadmin
DATABASE_PASSWORD=<from Vault>

# Messaging
AMQ_BROKER_URL=ssl://broker.mq.us-east-1.amazonaws.com:61617
AMQ_USERNAME=<from Vault>
AMQ_PASSWORD=<from Vault>

# Cache
REDIS_HOST=master.dev-cache-rg.8fxuvv.use1.cache.amazonaws.com
REDIS_PORT=6379

# Security

# Storage
AWS_S3_PUBLICURL=https://s3.amazonaws.com/embrix-public-dev
