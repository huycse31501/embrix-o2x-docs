Code of Conduct

Our Pledge

We are committed to providing a welcoming and inspiring community for all. By participating in this project, you agree to abide by our Code of Conduct.

Expected Behavior

  • Be respectful and inclusive
  • Welcome newcomers and help them get started
  • Accept constructive criticism gracefully
  • Focus on what is best for the project and community
  • Show empathy towards other community members

Unacceptable Behavior

  • Harassment, discrimination, or offensive comments
  • Personal attacks or trolling
  • Publishing others' private information
  • Other conduct that could reasonably be considered inappropriate

Getting Started

Prerequisites

Before you begin, ensure you have:
  • Java 8 (JDK 1.8.0_181+)
  • Maven 3.6+
  • Docker & Docker Compose
  • Git
  • Your favorite IDE (IntelliJ IDEA recommended)
  • Node.js 12+ (for frontend development)

Initial Setup

  1. Fork the repository:
       # Click "Fork" on GitHub
  2. Clone your fork:
       git clone https://github.com/YOUR_USERNAME/embrix-o2x.git
   cd embrix-o2x
  3. Add upstream remote:
       git remote add upstream https://github.com/embrix/embrix-o2x.git
  4. Set up infrastructure:
       # Start PostgreSQL, Redis, ActiveMQ, Vault
   docker-compose up -d
  5. Run database migrations:
       cd engine
   mvn flyway:migrate -Dspring.profiles.active=dev
  6. Build the project:
       # Build in correct order
   cd common && mvn clean install
   cd ../engine && mvn clean install
   # ... continue with other modules
  7. Run tests:
       mvn test

Development Workflow

Branch Naming Convention

Use descriptive branch names that follow this pattern: 
<type>/<ticket-number>-<short-description>

Examples:
feature/EP-1234-add-usage-rating
bugfix/EP-5678-fix-payment-allocation
hotfix/EP-9012-urgent-invoice-fix
refactor/EP-3456-simplify-billing-hub
docs/EP-7890-update-api-documentation
Types:
  • feature/ - New features
  • bugfix/ - Bug fixes
  • hotfix/ - Urgent production fixes
  • refactor/ - Code refactoring
  • docs/ - Documentation updates
  • test/ - Test improvements
  • chore/ - Build, CI/CD, or tooling changes

Typical Workflow

  1. Create a new branch:
       git checkout -b feature/EP-1234-add-usage-rating
  2. Make your changes:
    • Write code following coding standards

    • Add tests for new functionality

    • Update documentation as needed
  3. Commit your changes:
       git add .
   git commit -m "feat(usage): add tiered rating for voice calls"
  4. Keep your branch up to date:
       git fetch upstream
   git rebase upstream/develop
  5. Push to your fork:
       git push origin feature/EP-1234-add-usage-rating
  6. Create a Pull Request:
    • Go to GitHub and create a PR

    • Fill out the PR template

    • Link to the relevant issue/ticket

Coding Standards

Groovy Style Guide

Naming Conventions: 
// Classes: PascalCase
class CustomerHub { }

// Methods: camelCase
def processOrder() { }

// Variables: camelCase
def accountId = "ACC-001"

// Constants: UPPER_SNAKE_CASE
static final String DEFAULT_CURRENCY = "USD"

// Private fields: camelCase with _ prefix
private String _internalCache
Formatting: 
// Indentation: 4 spaces (not tabs)
class BillingHub {
    
    // Space after keywords
    if (condition) {
        // code
    }
    
    // Space around operators
    def total = price + tax
    
    // No space before method call parentheses
    def result = calculateTotal(amount, tax)
    
    // Space after commas
    def list = [1, 2, 3]
}
Best Practices: 
// Use def for local variables
def accountName = "TechCorp"

// Use explicit types for method parameters
String findAccountById(String accountId) {
    // ...
}

// Use @CompileStatic for performance-critical code
@CompileStatic
class UsageRatingService {
    // ...
}

// Null safety
def account = accountService.find(accountId)
if (account != null) {
    // safe to use
}

// Or use elvis operator
def name = account?.accountName ?: "Unknown"

Java Style Guide

Naming: 
// Classes: PascalCase
public class PaymentService { }

// Interfaces: PascalCase with "able" suffix or "I" prefix (avoid)
public interface Billable { }

// Methods: camelCase
public void processPayment() { }

// Constants: UPPER_SNAKE_CASE
public static final int MAX_RETRIES = 3;
Formatting: 
// Braces on same line
public class Account {
    private String accountId;
    
    public String getAccountId() {
        return accountId;
    }
}

// Line length: 120 characters maximum
// Use line breaks for long method chains
String result = accountService
    .findById(accountId)
    .map(Account::getAccountName)
    .orElse("Unknown");

JOOQ Patterns

// Good: Select specific fields
def accounts = dsl.select(
    ACCOUNT.ACCOUNT_ID,
    ACCOUNT.ACCOUNT_NAME,
    ACCOUNT.STATUS
)
.from(ACCOUNT)
.where(ACCOUNT.STATUS.eq(AccountStatus.ACTIVE))
.fetch()

// Good: Use type-safe conditions
def condition = ACCOUNT.CREATED_DATE
    .greaterOrEqual(startDate)
    .and(ACCOUNT.CREATED_DATE.lessThan(endDate))

// Good: Use fetch methods appropriately
def singleAccount = dsl.selectFrom(ACCOUNT)
    .where(ACCOUNT.ACCOUNT_ID.eq(accountId))
    .fetchOne()  // Returns null if not found

def accountList = dsl.selectFrom(ACCOUNT)
    .where(ACCOUNT.STATUS.eq(AccountStatus.ACTIVE))
    .fetch()  // Returns list (may be empty)

GraphQL Schema Conventions

# Types: PascalCase
type Account {
    # Fields: camelCase
    accountId: String!
    accountName: String
    status: AccountStatus!
    
    # Relations: camelCase (singular or plural)
    subscriptions: [Subscription]
    primaryContact: Contact
}

# Enums: PascalCase with UPPER_CASE values
enum AccountStatus {
    ACTIVE
    SUSPENDED
    CLOSED
}

# Mutations: verb + noun
type Mutation {
    createAccount(input: AccountInput!): Account
    updateAccount(accountId: String!, input: AccountInput!): Account
    deleteAccount(accountId: String!): Boolean
}

# Queries: noun or verb + noun
type Query {
    account(accountId: String!): Account
    searchAccounts(filter: AccountFilter): AccountSearchResult
}

React/JavaScript Style Guide

// Components: PascalCase
const AccountList = () => {
    // Hooks at the top
    const [accounts, setAccounts] = useState([]);

Testing Requirements

Test Coverage Targets

  • Unit Tests: 75-85% coverage
  • Integration Tests: Critical paths covered
  • E2E Tests: Major user journeys

Unit Tests (Spock)

import spock.lang.Specification

class BillingHubSpec extends Specification {
    
    BillingHub billingHub
    AccountService accountService = Mock()
    
    def setup() {
        billingHub = new BillingHub(accountService: accountService)
    }
    
    def "should calculate prorated charge correctly"() {
        given: "an account with mid-cycle subscription"
        def account = new Account(accountId: "ACC-001")
        def subscription = new Subscription(
            startDate: LocalDate.of(2026, 2, 15),
            monthlyCharge: 100.00
        )
        
        when: "proration is calculated"
        def proratedAmount = billingHub.calculateProration(
            subscription, 
            LocalDate.of(2026, 2, 1)
        )
        
        then: "amount is prorated for partial month"
        proratedAmount == 50.00  // 15 days out of 30
        
        and: "account service was called"
        1 * accountService.find("ACC-001") >> account
    }
    
    def "should throw exception when account not found"() {
        given: "non-existent account"
        accountService.find("ACC-999") >> null
        
        when: "billing is attempted"
        billingHub.runBilling("ACC-999")
        
        then: "exception is thrown"
        thrown(AccountNotFoundException)
    }
}

Integration Tests

@SpringBootTest
@Testcontainers
class OrderProcessingIntegrationTest {
    
    @Container
    static PostgreSQLContainer postgres = new PostgreSQLContainer("postgres:10.5")
    
    @Autowired
    OrderProcessService orderProcessService
    
    @Autowired
    DSLContext dsl
    
    @BeforeEach
    void setup() {
        // Clean test data
        dsl.deleteFrom(ORDER).execute()
    }
    
    @Test
    void shouldProcessOrderEndToEnd() {
        // given
        def order = createTestOrder()
        
        // when
        def result = orderProcessService.process(order)
        
        // then
        assertThat(result.status).isEqualTo(OrderStatus.PROVISIONING_INITIATED)
        
        // verify database state
        def dbOrder = dsl.selectFrom(ORDER)
            .where(ORDER.ORDER_ID.eq(order.orderId))
            .fetchOne()
        assertThat(dbOrder).isNotNull()
    }
}

Test Naming

// Pattern: should_expectedBehavior_when_stateUnderTest

def "should throw exception when account is suspended"() { }

def "should create subscription when order is completed"() { }

Running Tests

# All unit tests
mvn test

# Specific test class
mvn test -Dtest=BillingHubSpec

# Integration tests
mvn verify -Pintegration-tests

# Skip tests (use sparingly)
mvn install -DskipTests

Review Process

  1. Automated Checks:
    • Build must pass

    • Tests must pass

    • Code coverage must not decrease
  2. Code Review:
    • At least 1 approval required

    • All comments must be addressed

    • No unresolved conversations
  3. Merge:
    • Squash and merge (preferred)

    • Rebase and merge (for feature branches)

    • Create merge commit (for release branches)

Format

Documentation

Update documentation when you:
  • Add new features
  • Change existing functionality
  • Add new API endpoints
  • Modify database schema
  • Change configuration
  • Fix important bugs

Documentation Types

  1. Code Comments:

   BigDecimal calculateProration(Subscription subscription, LocalDate effectiveDate) {

   }

    • Each module should have a README

    • Explain module purpose and usage

    • Include examples
  3. API Documentation:
    • Update GraphQL schema comments

    • Add examples to API_REFERENCE.md

    • Update Swagger annotations for REST APIs

    • Update COMPREHENSIVE_ARCHITECTURE.md for major changes

    • Update diagrams when architecture changes

    • Keep NEWCOMER_GUIDE files current

Code Review Guidelines

For Authors

  • Keep PRs small and focused (< 400 lines if possible)
  • Provide context in PR description
  • Respond to feedback promptly
  • Don't take feedback personally

For Reviewers

  • Be respectful and constructive
  • Focus on the code, not the person
  • Explain reasoning behind suggestions
  • Approve when code meets standards

What to Look For

  • Correctness: Does the code do what it's supposed to?
  • Testing: Are there adequate tests?
  • Design: Is the design sound?
  • Readability: Is the code easy to understand?
  • Performance: Are there performance concerns?
  • Security: Are there security vulnerabilities?

Review Checklist

  • [ ] Code follows style guidelines
  • [ ] Logic is correct and handles edge cases
  • [ ] Tests are comprehensive
  • [ ] No obvious performance issues
  • [ ] No security vulnerabilities
  • [ ] Documentation is updated
  • [ ] Error handling is appropriate
  • [ ] Logging is appropriate
  • [ ] No hardcoded values (use config)
